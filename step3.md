# Step 3: HTTP POST 요청 처리, 데이터 바인딩 및 PRG 패턴 (Redirect)

이 가이드는 스프링 MVC에서 HTTP POST 요청을 받아 사용자 입력 데이터를 객체(DTO)로 자동 바인딩하는 방법, 세션(Session)을 이용한 데이터 임시 보관, 그리고 새로고침 시 데이터가 중복 제출되는 문제를 막는 PRG(Post-Redirect-Get) 패턴의 원리를 설명합니다.

---

## 1. 🐣 초심자를 위한 비유로 이해하기

스프링 웹 MVC의 데이터 입력과 페이지 이동 방식을 **"스마트 호텔의 주문 서비스"**에 비유해 볼 수 있습니다.

| 개념 | 호텔에서의 비유 | 설명 |
| :--- | :--- | :--- |
| **`@ModelAttribute`** (커맨드 객체) | **자동 짐 정리 로봇** | 손님이 캐리어(Form 데이터)에 여러 물건을 무작위로 담아 보냈을 때, 지배인이 일일이 손으로 꺼내는 대신(`@RequestParam`), 정리 로봇(`@ModelAttribute`)이 DTO 규격에 맞춰 물건들을 상자(`ShirtDTO`)에 자동으로 분류하여 담아주는 방식입니다. |
| **`HttpSession`** (세션) | **호텔 개인 금고 (사물함)** | 손님이 체크아웃하거나 방을 옮기기 전까지 계속 유지되는 사물함입니다. 잠시 다른 부서(리다이렉트)를 다녀와도 이 사물함에 보관해 둔 데이터(`shirt`)는 그대로 찾아서 꺼내 쓸 수 있습니다. |
| **Forward (포워드)** | **호텔 내부 비밀 이동** | 손님이 모르게 지배인이 직접 손님 가방을 들고 뒷문으로 옆 부서(JSP)에 가서 일을 처리해 주는 것입니다. 손님은 방을 옮겼는지 알 수 없습니다(브라우저 URL이 안 바뀜). |
| **Redirect (리다이렉트)** | **"정문으로 다시 오세요" 초대장** | 손님의 주문을 접수한 뒤, "접수 완료되었으니 저기 `/shirt` 정문으로 새로 오셔서 대기해 주세요" 하고 **302 새 대기표**를 쥐여주는 것입니다. 손님(브라우저)은 이 대기표를 보고 주소창을 새로 치고 들어오게 됩니다. |
| **PRG 패턴** (Post-Redirect-Get) | **중복 주문 방지 안심벨** | 포스트기로 주문을 넣고(POST), 바로 주문 확인판으로 화면을 돌려주어(Redirect), 손님이 새로고침을 눌러도 주문이 두 번 들어가지 않고 확인 화면만 계속 새로 조회(GET)되게 만드는 장치입니다. |

## 2. 💻 주니어를 위한 핵심 원리 설명

### ① `@ModelAttribute`를 이용한 커맨드 객체 바인딩 (Data Binding)
```java
@PostMapping
public String form(@ModelAttribute ShirtDTO shirtDTO, HttpSession session) {
    session.setAttribute("shirt", shirtDTO);
    return "redirect:/shirt";
}
```
* **동작 원리**: 클라이언트가 form을 통해 보낸 파라미터(`name`, `price`)를 스프링 MVC가 자동으로 `@ModelAttribute` 대상 객체(`ShirtDTO`)에 채워줍니다.
* **내부 메커니즘**:
  1. `HandlerAdapter`가 요청 데이터를 바인딩하기 위해 `@ModelAttribute` 클래스의 기본 생성자(`@NoArgsConstructor`)를 찾아 인스턴스를 생성합니다.
  2. 스프링의 `WebDataBinder`가 HTTP 요청 파라미터의 이름(Form 태그의 `name="price"`)과 DTO의 프로퍼티 이름이 일치하는지 비교합니다.
  3. 일치하는 경우 자바빈즈 프로퍼티 규약(Setter 메서드: `setPrice()`)에 의거해 값을 주입합니다.
  4. 이때 문자열로 전송된 데이터 `"12000"`을 자바의 `int` 타입으로 형변환하는 작업 역시 스프링의 `ConversionService`가 자동으로 수행합니다.

### ② Lombok 어노테이션 확장
* **`@ToString` 추가**: DTO 클래스에 `@ToString`을 적용하여 `System.out.println("shirtDTO = " + shirtDTO)`로 로깅했을 때 클래스명과 내부 변수 상태(`ShirtDTO(name=..., price=...)`)를 가독성 있는 문자열로 바로 출력되도록 지원합니다.

### ③ `HttpSession`을 활용한 데이터 유지
* **스코프의 이해**: 컨트롤러 핸들러 메서드에 `HttpSession` 파라미터를 추가하면, 스프링이 현재 요청의 `HttpServletRequest.getSession()`을 호출하여 세션 객체를 주입합니다.
* **데이터 유지**: POST 요청에서 가공하거나 생성한 데이터(`ShirtDTO`)를 세션 영역(`session.setAttribute("shirt", shirtDTO)`)에 보관해 두면, 이후 리다이렉트되어 발생하는 완전히 새로운 HTTP GET 요청에서도 해당 데이터에 접근할 수 있어 JSP(`${shirt}`)가 정보를 정상적으로 출력하게 됩니다.

### ④ Forward vs Redirect 및 PRG 패턴 (Post-Redirect-Get)
* **Forward (포워드)**: 
  * 서버 내부에서의 흐름 이동입니다. 웹 브라우저는 단 한 번의 HTTP 요청만 보냈고, 서버가 내부적으로 서블릿에서 JSP 파일로 포워드(전달)한 것이므로 브라우저의 URL 주소창은 변하지 않습니다.
  * 만약 POST 요청 후 포워드로 JSP를 보여준 상태에서 사용자가 **새로고침(F5)**을 누르면, 브라우저는 가장 마지막에 보낸 HTTP POST 요청을 재전송합니다. 이로 인해 동일한 데이터가 데이터베이스에 이중으로 저장되는 **Double Submit** 문제가 발생합니다.
* **Redirect (리다이렉트)**:
  * 리턴 값으로 `"redirect:/shirt"`를 보내면, 스프링 MVC는 HTTP 응답 헤더로 `Status 302 Found`와 `Location: /shirt`를 돌려줍니다.
  * 브라우저는 이 응답을 받자마자 지정된 주소(`/shirt`)로 새로운 HTTP **GET** 요청을 스스로 다시 날립니다.
* **PRG 패턴**: 
  * `POST(등록 요청)` -> `Redirect(302 응답)` -> `GET(상세 화면 조회)` 구조를 이룸으로써 사용자가 새로고침을 해도 안전하게 GET 요청만 재전송되도록 보장합니다.

## 3. 🎯 면접 대비 예상 질문 & 모범 답변

### Q1. `@RequestParam`과 `@ModelAttribute`의 쓰임새와 동작 원리의 차이점을 설명해 주세요.
> **답변:** 
> * **`@RequestParam`**은 HTTP 요청 파라미터(Query String 또는 Form 데이터)를 1:1로 개별 자바 기본 타입이나 객체 타입으로 매핑할 때 사용합니다.
> * **`@ModelAttribute`**는 여러 개의 요청 파라미터를 자바 객체(DTO/커맨드 객체)의 필드 구조에 맞게 통째로 바인딩(Binding)할 때 사용합니다. 내부적으로 대상 객체의 기본 생성자를 호출해 인스턴스를 만들고, 파라미터 이름과 일치하는 프로퍼티의 Setter 메서드를 호출하여 값을 주입하며, 필요한 타입 변환까지 자동으로 처리해 줍니다.

### Q2. PRG(Post-Redirect-Get) 패턴이란 무엇이며, 왜 사용해야 하나요?
> **답변:** 
> PRG 패턴은 HTTP POST 요청 처리 이후 뷰를 곧바로 포워드(Forward)하는 대신, 브라우저를 다른 GET URL로 리다이렉트(Redirect)시키는 디자인 패턴입니다.
> 만약 POST 요청 후 단순히 포워드하여 화면을 렌더링하면, 사용자가 브라우저에서 '새로고침'을 눌렀을 때 마지막에 보낸 POST 요청이 중복 전송되어 결제 중복이나 게시글 이중 등록 등의 오류(Double Submit)가 생깁니다. 리다이렉트를 수행하면 브라우저의 주소창이 새로운 GET 경로로 바뀌기 때문에, 새로고침을 해도 안전하게 조회(GET) 요청만 반복되어 중복 등록 문제를 미연에 방지할 수 있습니다.

### Q3. Redirect와 Forward의 차이를 HTTP 및 브라우저 관점에서 설명해 주세요.
> **답변:** 
> * **Forward**는 서버 내부에서 일어나는 경로 이동으로, 클라이언트는 이동 여부를 모릅니다. 단 한 번의 HTTP 요청/응답 사이클만 일어나며, 브라우저의 주소창 URL은 변하지 않고 최초 요청 주소로 유지됩니다.
> * **Redirect**는 서버가 브라우저에게 "이 주소로 다시 요청하라"는 HTTP 302 응답 코드와 Location 헤더를 주는 방식입니다. 이에 브라우저는 새로운 URL로 새로운 HTTP 요청을 전송하게 되어 총 두 번의 요청/응답 사이클이 발생하며, 브라우저 주소창의 URL도 새 주소로 변경됩니다.

### Q4. `@ModelAttribute` 객체를 사용할 때 DTO에 기본 생성자(`@NoArgsConstructor`)와 Setter(`@Setter`) 메서드가 필요한 이유는 무엇인가요?
> **답변:** 
> 스프링 핸들러 어댑터가 파라미터의 `@ModelAttribute`를 처리할 때, 가장 먼저 인스턴스를 동적으로 생성하기 위해 리플렉션을 통해 DTO의 기본 생성자를 먼저 호출하기 때문입니다. 생성 후에는 요청 파라미터 이름(예: `name`)과 매칭되는 Setter 메서드(예: `setName()`)를 호출하여 값을 바인딩합니다. 따라서 기본 생성자와 자바빈즈 규약에 따른 Setter 메서드가 존재해야 정상적으로 데이터 바인딩 프로세스가 완료될 수 있습니다.
