# Step 4: Web Bean Scopes (Request Scope vs Session Scope) & Scoped Proxy

이 가이드는 스프링 컨테이너가 제공하는 웹 애플리케이션 환경의 다양한 빈 스코프(Bean Scope)인 Request Scope와 Session Scope의 개념, 싱글톤 빈과 웹 스코프 빈 결합 시 발생하는 한계점 및 이를 해결하는 프록시(Proxy) 기법을 설명합니다.

---

## 1. 🐣 초심자를 위한 비유로 이해하기

스프링 빈의 주입과 스코프 개념을 **"호텔의 공용 시계, 영수증, 그리고 카드키"**에 비유해 볼 수 있습니다.

| 개념 | 호텔에서의 비유 | 설명 |
| :--- | :--- | :--- |
| **`Singleton`** (컨트롤러) | **호텔 로비의 공용 대형 시계** | 호텔 전체에 단 하나만 존재하며, 모든 직원이 똑같은 시계를 보고 동일한 시간 정보를 공유합니다. |
| **`@RequestScope`** (요청 스코프) | **일회용 주문 영수증** | 손님이 주문할 때마다 매번 새로 뽑아서 쓰는 영수증입니다. 주문이 처리되면 바로 버려지고(소멸), 다음 요청이 들어오면 완전히 새로운 영수증(새 인스턴스)이 발급됩니다. |
| **`@SessionScope`** (세션 스코프) | **투숙객 전용 객실 카드키** | 손님이 체크인하여 머무는 동안(세션 유지 기간)에는 계속 가지고 다니는 유일한 카드키입니다. 다른 부서(새 요청)를 갔다 와도 똑같은 카드키를 사용하지만, 다른 손님이 오면 그 손님 전용 카드키가 새로 발급됩니다. |
| **`Scoped Proxy`** (마법 폴더) | **현재 손님의 영수증 연결 폴더** | 싱글톤 지배인(`ScopeController`)은 한 번 채용되면 안 바뀝니다. 지배인 손에 특정 손님의 일회용 영수증을 쥐여주면 다음 손님이 왔을 때 엉뚱한 영수증을 보게 됩니다. 이를 막기 위해 지배인에게 **"현재 요청 중인 진짜 손님의 영수증으로 연결되는 마법 폴더(Proxy)"**를 쥐여주는 것입니다. 지배인이 폴더를 열 때마다 폴더는 지금 로비에 서 있는 손님의 영수증을 알아서 짠 하고 띄워줍니다. |

---

## 2. 💻 주니어를 위한 핵심 원리 설명

### ① 다양한 스프링 빈 스코프 (Spring Bean Scopes)
스프링 빈은 정의된 수명 주기(Lifecycle)를 가지며 이를 스코프라고 부릅니다.
* **Singleton (기본값)**: 애플리케이션 구동 시 컨테이너당 단 하나의 인스턴스만 생성되어 공유됩니다.
* **Prototype**: 컨테이너에 빈을 요청할 때마다 항상 새로운 인스턴스를 생성해 반환합니다. 생성 이후 소멸 주기는 개발자가 직접 관리해야 합니다.
* **Request Scope (`@RequestScope`)**: HTTP 요청 하나가 시작되어 끝날 때까지 유효한 스코프입니다. 각 HTTP 요청마다 별도의 빈 인스턴스가 생성되고 요청이 끝나면 소멸합니다.
* **Session Scope (`@SessionScope`)**: HTTP Session과 동일한 생명 주기를 갖는 스코프입니다. 하나의 웹 브라우저(사용자 세션)당 하나의 인스턴스가 할당되며 세션 만료 시 소멸합니다.

### ② 생성자 주입과 Lombok `@RequiredArgsConstructor`
```java
@Controller
@RequestMapping("/scope")
@RequiredArgsConstructor
public class ScopeController {
    private final RequestScopeBean requestScopeBean;
    private final SessionScopeBean sessionScopeBean;
    ...
}
```
* **Lombok 역할**: `@RequiredArgsConstructor` 어노테이션은 클래스 내의 `final` 필드나 `@NonNull`이 붙은 필드를 매개변수로 갖는 생성자를 컴파일 시점에 자동으로 만들어 줍니다.
* **스프링의 자동 생성자 주입**: Spring 4.3 버전부터 클래스의 생성자가 단 하나만 존재한다면, `@Autowired` 어노테이션을 생략해도 스프링이 이 생성자를 통해 필요한 빈들을 자동으로 주입(Constructor Injection)해 줍니다. 

### ③ 웹 스코프 빈의 주입과 프록시 패턴 (Scoped Proxy)
* **근본적 딜레마**: `ScopeController`는 싱글톤(Singleton) 빈입니다. 즉, 애플리케이션 시작 단계에 단 한 번만 생성되어 고정됩니다. 그러나 `RequestScopeBean`은 HTTP 요청이 들어올 때 비로소 생성되는 빈입니다. 애플리케이션 구동 시점에는 아직 요청이 들어오지 않았기 때문에 주입할 진짜 인스턴스 자체가 존재하지 않는 모순이 발생합니다.
* **AOP Scoped Proxy 메커니즘**:
  * 스프링은 이 모순을 해결하기 위해 진짜 객체 대신 **가짜 프록시(Proxy) 객체**를 주입합니다.
  * `@RequestScope` 또는 `@SessionScope`가 붙은 빈을 컴파일/런타임 시점에 분석하여, 스프링은 내부적으로 CGLIB 바이트코드 조작 라이브러리를 활용해 대상 클래스를 상속받은 가짜 프록시 객체를 동적으로 생성하여 빈 저장소에 보관합니다.
  * 컨트롤러가 주입받은 객체는 이 **가짜 프록시 객체**입니다.
  * 컨트롤러 내에서 `requestScopeBean.getUuid()`를 호출하면, 가짜 프록시 객체가 실행 시점에 현재 쓰레드와 연결된 HTTP 요청 정보(`RequestContextHolder`)에서 진짜 `RequestScopeBean` 인스턴스를 조회하여 메서드 호출을 위임(Delegate)합니다.

## 3. 🎯 면접 대비 예상 질문 & 모범 답변

### Q1. 스프링 빈의 `@RequestScope`와 `@SessionScope`에 대해 설명해 주시고 각각 주로 언제 활용되는지 알려주세요.
> **답변:** 
> * **`@RequestScope`**는 하나의 HTTP 요청 주기 동안만 생존하는 스코프입니다. 주로 개별 요청 내에서 클라이언트의 API 호출 로깅이나 트래킹용 임시 UUID 보관, 또는 요청 처리에 특화된 컨텍스트 데이터를 전파할 때 유용하게 쓰입니다.
> * **`@SessionScope`**는 웹 브라우저당 생성되는 HTTP 세션의 생명 주기와 일치하는 스코프입니다. 주로 로그인한 사용자의 고유 정보, 쇼핑몰의 장바구니 리스트 등 사용자가 브라우저를 닫거나 세션이 만료되기 전까지 여러 요청에 걸쳐 유지해야 하는 사용자 개인 상태 값을 저장할 때 사용합니다.

### Q2. 싱글톤 빈인 컨트롤러에 웹 스코프(Request, Session) 빈을 주입할 때 왜 프록시(Proxy) 설정이 필요한가요?
> **답변:** 
> 싱글톤 빈은 애플리케이션 초기 구동 시점에 컨테이너에 의해 단 한 번만 생성되고 의존관계 주입이 종료됩니다. 하지만 Request 스코프 빈 등은 실제 웹 요청이 들어온 시점에 개별적으로 생성되므로, 컨트롤러 생성 단계에서는 주입할 인스턴스가 존재하지 않습니다. 또한 단순 주입을 해버리면 싱글톤 컨트롤러가 최초에 생성된 웹 스코프 빈 하나를 영원히 고정으로 참조하므로 요청마다 새로 인스턴스가 생성되는 스코프 본연의 목적이 훼손됩니다.
> 따라서 프록시 설정을 통해 컨트롤러에는 진짜 빈을 우회 참조하는 **가짜 프록시 객체**를 주입해 두고, 실제 메서드가 호출되는 시점에 현재 요청(ThreadLocal)에 해당되는 진짜 인스턴스를 찾아서 대행하도록 처리해야 합니다.

### Q3. 생성자 주입(Constructor Injection)을 사용할 때 Lombok의 `@RequiredArgsConstructor`를 결합하면 얻는 이점은 무엇인가요?
> **답변:** 
> 생성자 주입은 필드 주입이나 Setter 주입과 달리 객체 생성 시점에 의존성이 완전히 바인딩되므로 **객체의 불변성(Immutable)**을 보장하고, 테스트 코드 작성 시 순수 자바 코드로 목(Mock) 객체를 주입하기 편리하여 가장 권장되는 주입 방식입니다.
> 여기에 Lombok의 `@RequiredArgsConstructor`를 사용하면, 주입받을 협력 객체 필드를 `final`로 선언하는 것만으로 보일러플레이트 코드인 매개변수 생성자를 컴파일러가 알아서 생성해 주므로 가독성이 높아지고 생산성이 크게 향상됩니다.

### Q4. 프록시 객체가 런타임에 실제 스코프 빈의 인스턴스를 찾아내는 동작 방식을 서블릿 컨텍스트 수준에서 간략히 설명해 주세요.
> **답변:** 
> 스프링이 생성한 가짜 프록시 빈은 내부적으로 **`RequestContextHolder`**를 참조합니다. 스프링 웹 애플리케이션에서는 모든 HTTP 요청이 들어올 때 `DispatcherServlet` 앞단에서 현재 HTTP 요청과 응답을 쓰레드 로컬(ThreadLocal) 저장소 기반인 `RequestContextHolder`에 등록합니다. 프록시 객체의 메서드가 호출되면, 프록시는 이 `RequestContextHolder`로부터 현재 동작 중인 쓰레드의 서블릿 요청 객체를 조회하고, 이와 매핑된 해당 요청 스코프 컨텍스트 또는 세션 컨텍스트에서 진짜 빈 인스턴스를 찾아 메서드 호출을 포워딩해 줍니다.
